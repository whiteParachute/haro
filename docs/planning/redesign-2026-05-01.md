## 0. 触发背景

owner 在 2026-05-01 提出"单人难以快速适配当前 spec 全速冲刺"，要求与 [happyclaw / AgentDock](https://github.com/ar8327/happyclaw) 做横向对比并大幅精简。经过 4 轮往返修正，定下当前重设计方案。

参考项目：
- `ar8327/happyclaw`（AgentDock 分支） — 自托管单用户多会话 agent workbench；Haro 的 workbench 层借鉴对象
- `hermes-agent` — CLI 优先、配置可在终端完成；Haro 的 CLI 风格借鉴对象
- `hermes-web-ui` — Web UI 与后端解耦的范例；Haro 的前后端解耦借鉴对象

## 1. 修正后的项目定位（双层架构）

```
┌─────────────────────────────────────────────────┐
│  进化层（Haro 真正的差异化）                     │
│  - 使用记忆驱动（Self-Monitor）                  │
│  - 业界趋势驱动（Industry Intel — 新模块）       │
│  - 用户决策驱动（Evolution Proposal/Approval）   │
│  - Agent 自判断驱动（Pattern Miner + 自规划）    │
└─────────────────────────────────────────────────┘
                      ↑ 喂数据
┌─────────────────────────────────────────────────┐
│  Workbench 层（happyclaw 级别的日用底座）        │
│  - Web Channel + MCP 工具层 + 定时 + 流式 UX     │
│  - 多 Agent 编排 + Memory Fabric + 预算          │
└─────────────────────────────────────────────────┘
```

**关键认知**：进化层寄生在 workbench 之上——没有日用底座产生使用数据，进化引擎就没东西可吃。所以"先 workbench、后进化"是必然顺序，不是对原愿景的妥协。

## 2. 四条边界约束

1. **CLI 优先**：CLI 是"功能等价 Web UI 减去图形化体验"的完整入口，必须能在脱离 Web UI 的情况下完成全部使用与配置（hermes-agent 风格）
2. **Web UI 与后端解耦**：Web 前端通过 OpenAPI/JSON schema 与 Web API 通信，前后端独立 `package.json`、可独立发布（hermes-web-ui 风格）
3. **多 provider 抽象保留**：当前唯一实现是 Codex，但抽象层完整保留，规划接入 xiaomi-token-plan / kimi-token-plan 等
4. **多 channel 抽象保留**：CLI / 飞书 / Telegram 全保留；规划新增 Web channel；后续可能接入更多

## 3. 自用前提（owner 是自用单人，但功能保留）

- 单人自用，但**多用户能力保留**（仅降优先级，不主动打磨）
- 数据量未必小，**分页框架保留**
- **i18n / Audit / Bootstrap / Gateway / 并行预算汇总 / systemd 检查** 全部保留
- 不是产品级对外发布

## 11. 关键背景（为什么这么定）

- happyclaw 是单用户多会话 workbench，已经在 UX 抛光、4 channel、双 runner、定时任务、Web 终端、PWA、流式 UX 上领先
- Haro 在多 Agent 编排、eat/shit 代谢、权限预算、Memory Fabric 上有 happyclaw 没有的差异化
- owner 自用单人 + 单 Codex provider + 不要 QQ/微信 + 必须加 Web channel + CLI 优先 + 前后端解耦
- 进化层是核心愿景，不能砍；但需要 workbench 先稳定才能产生供进化层消化的数据
- 现有所有已实现功能保留（除了 Web Server 要从 CLI 包剥离这一项架构调整）
