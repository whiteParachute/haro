# 多 Agent 设计约束规范（强制）

**状态：强制执行。所有 Haro 内部 Agent 编排和 Team Orchestrator 的实现必须遵守本规范。**

## 背景

本规范基于以下分析制定：
1. SagaSu《三省六部幻觉：为什么"虚拟公司"式多 Agent 架构在工程上不成立》（2026-04-14）
2. Anthropic / OpenAI / Google 三大厂商的多 Agent 工程实践
3. Haro 架构设计与上述分析的对照检验

核心论点："虚拟公司式多 Agent 架构"（按人类岗位划分 Agent + 流水线式单向交接 + 摘要压缩传递）在工程上不成立，因为它系统性地破坏了推理链的连续性。

## 五条强制约束

### 约束①：传原文者活（Pass the Source, Not the Summary）

**规则**：下游 Agent 必须接收上游的原始材料，禁止接收上游 Agent 的理解、摘要或结论。

**合规设计**：
- Memory Fabric 共享原始上下文，所有 Agent 从共享存储读取原始数据
- `evolution-context/` 共享目录存储原始数据，不压缩传递
- 使用 Checkpointing 保存完整状态，而非关键信息摘要

**违规示例（禁止）**：
```
Agent A 分析代码 → 输出"代码质量较差，有3个问题" → Agent B 根据此摘要修复
```

**合规示例**：
```
Agent A 分析代码 → 把原始代码 + 完整分析结果写入共享存储
Agent B 从共享存储读取原始代码 + Agent A 的完整分析（非摘要）→ 修复
```

### 约束②：推理链分叉再合并（Fork and Merge, Never Chain）

**规则**：多 Agent 协作必须采用 hub-spoke 拓扑（分叉再合并），禁止串行交接链（chain）。

**合规拓扑**：
```
        ┌→ Agent A（探索方向 1）→┐
Orchestrator                      → Orchestrator（合并）
        └→ Agent B（探索方向 2）→┘
```

**违规拓扑（禁止）**：
```
Agent A → Agent B → Agent C → Agent D（流水线）
```

**Scenario Router 要求**：图模型必须确保所有分支结果回流到同一编排节点后再进行下一步。

### 约束③：并行覆盖不是分工（Parallel Coverage, Not Role Division）

**规则**：多 Agent 的价值在于扩大搜索空间（并行探索多个可能性），而不是模拟人类组织分工。

**Team 拆分维度**：信息属性 / 搜索空间 / 假设维度，而非人类组织角色。

**合规拆分示例**：
- "Agent A 探索方案 X，Agent B 探索方案 Y，Agent C 评估两者" ✅
- "Agent A 搜索互联网信息，Agent B 搜索本地代码库" ✅

**违规拆分示例（禁止）**：
- "Agent A 是 PM，Agent B 是开发，Agent C 是测试" ❌
- "Agent A 负责编码，Agent B 负责测试" ❌

### 约束④：验证 Agent 是否定者（Validator is an Adversary）

**规则**：验证 / 评估 Agent 的职责是对抗性地找问题，而不是接棒执行下一步任务。

**合规设计**：
- 验证 Agent 接收原始方案，寻找漏洞、错误、边界案例
- 验证 Agent 输出"否定清单"（哪里有问题），而非"确认通过"
- Evolution Engine 的验证门控必须包含对抗性验证 Agent

**违规设计（禁止）**：
- 验证 Agent 只做"格式检查"然后传递给下一步 ❌
- 验证 Agent 看到方案后直接做改进实现 ❌

**Evolution Engine 特别要求**：三阶段循环（评估→规划→实现）中，评估阶段的验证 Agent 必须能访问完整原始状态，其输出是问题列表，规划阶段不得将其压缩为"评估通过/失败"结论。

### 约束⑤：工具是工具不是角色（Tools Define Capability, Not Role）

**规则**：Agent 的能力由其绑定的工具决定，不由角色标签限制。禁止创建"只能做某类事"的角色化 Agent。

**合规设计**：
- Agent 通过 `tools` 字段声明有哪些工具可用
- 同一个 Agent 可以在不同任务中使用不同工具子集
- ContextPlugin + 装饰器链按需绑定工具，不预设角色

**违规设计（禁止）**：
- 创建"CodeAgent"只能写代码，"SearchAgent"只能搜索 ❌
- 用角色标签限制 Agent 行为，而非工具权限 ❌

## 额外约束

### Pipeline 使用限制

**规则**：Pipeline 模式仅限确定性工具链（输入→工具→输出，无推理分支）。

**合规使用**：数据清洗流水线、格式转换、日志聚合等纯工具链。

**禁止使用 Pipeline 的场景**：任何涉及推理、判断、分析的场景。

### 跨 Session 状态文件

**规则**：所有跨 session 的 Agent 必须维护包含以下四类信息的状态文件：

1. **任务上下文**：当前任务描述、目标、约束
2. **执行历史**：已完成的操作记录（非摘要，而是结构化事件）
3. **关键决策**：重要选择及其依据（完整推理链）
4. **未完成工作**：待处理的 todo 和阻塞项

状态文件存储于 `~/.haro/agents/{name}/state.json`，格式为 JSON。

### Evolution Engine 阶段间信息流

**规则**：Evolution Engine 各阶段（评估→规划→实现）必须能访问前序阶段的完整原始数据。

**实现要求**：
- `evolution-context/` 目录作为各阶段的共享数据存储
- 每个阶段写入完整数据（含推理过程），不仅是结论
- 后续阶段从 `evolution-context/` 读取，不依赖前一阶段的输出摘要

## 合规检验矩阵

| 设计决策 | 检验问题 | 合规标准 |
|---------|---------|---------|
| Agent 划分 | 是否按角色/岗位划分？ | 否（按工具/信息维度） |
| 信息传递 | 是否传递摘要/结论？ | 否（传原始数据） |
| 协作拓扑 | 是否串行交接？ | 否（hub-spoke） |
| 验证设计 | 验证 Agent 是在找问题还是接棒？ | 找问题（否定者） |
| 能力定义 | 是否用角色标签限制 Agent？ | 否（工具绑定定义能力） |
